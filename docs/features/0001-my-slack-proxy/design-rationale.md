# 0001-my-slack-proxy — DESIGN RATIONALE

## Decision-1: routing-workflow-not-autonomous-agent
Forces: the SPEC contract is a small set of *distinct, enumerable* branches (ignore / auto-reply / surface / escalate), and the dominant risk is acting wrongly on someone's behalf. Alternative weighed: a fully autonomous agent loop that decides its own steps per message. Rejected because autonomy trades latency and cost for capability and invites *compounding errors* on a path that posts to colleagues — exactly where reliability matters most; the branching here is known in advance, so a predefined routing workflow is the least-autonomous structure that meets the requirement (Anthropic, *Building Effective Agents*; LangChain agency-spectrum). Agency is kept where it genuinely helps — the bounded context-gather + draft inside one branch — and nowhere else. This also makes every decision loggable and replayable for the eval gates (Decision-8). Invalidation trigger: if real traffic shows the four branches are not cleanly separable (frequent mis-routes that no classifier tuning fixes), revisit toward a state-machine/agentic triage — but measure escalation/false-filter rates first rather than adding agency "on vibes."

## Decision-2: socket-mode-ingestion
Forces: the deployment must hold no public inbound surface (security posture), and must observe continuously. Alternative weighed: Events API with a public HTTPS webhook. Rejected because it requires a publicly reachable, signed endpoint — the inbound attack surface we explicitly want to avoid on a personal/early deployment. A second, decisive force surfaced in research: Slack tightened 2025 rate limits on `conversations.history`/`replies` for non-Marketplace apps to roughly 1 req/min — so a polling design is effectively dead on arrival, and the event stream is the only viable continuous-ingestion path regardless. Caveat carried: Socket Mode is positioned for internal/single-workspace apps and is not for Marketplace distribution, and has a small per-app connection cap — fine for this internal tool. Invalidation trigger: distributing via the Slack Marketplace, or scaling past the connection cap, would force Events API + Marketplace approval (and re-open the rate-limit question for any history backfill). `[VERIFY]` exact 2025 limits and connection cap when the research quota resets.

## Decision-3: user-token-read-bot-token-disclosed-post
Forces: the proxy must see what *you* see (including DMs and private channels a bot can't), yet must never be mistaken for you (`SPEC#INV-2`). A bot token only sees channels the bot is invited to — insufficient for "uncaught-but-relevant" coverage. So reading requires the principal's own user token. Posting is the opposite concern: a user token would post *as the human* with no badge — true impersonation, forbidden. The clean split — read with user token, write with bot token — lands disclosure as a *platform-enforced* property: Slack stamps every bot-token message with an APP badge that even `chat:write.customize` cannot remove. That is far stronger than relying on the model to add a disclaimer. "On behalf of @user" is conveyed in copy + thread placement, keeping authorship visibly the app. Invalidation trigger: if Slack ever let bot messages drop the APP badge, `INV-2` would need an application-level disclosure guarantee instead.

## Decision-4: mcp-context-adapter
Forces: "as informed as you" spans many heterogeneous sources (Confluence/Jira, GitHub, Workspace, Slack), and the app must stay org-agnostic/portable. Alternative weighed: bespoke per-source API clients. Rejected because it hard-couples the core to each vendor's SDK and re-implements auth/tooling N times; MCP is the current de-facto open standard (Anthropic 2024; broad cross-vendor adoption) that lets each source be a swappable server behind one protocol, which *is* the org-adapter seam the security posture asked for. Accepted caveats: MCP is fast-moving (pin server versions) and its auth story (OAuth 2.1, confused-deputy and prompt-injection-via-tool risks) is the real design risk — so per-source credential scoping and per-principal isolation are treated as first-class (Decision-5), not assumed. Invalidation trigger: if a required source has no usable MCP server and writing one is uneconomical, that single source can fall back to a direct client behind the same adapter interface. `[VERIFY]` current MCP spec/auth version when quota resets.

## Decision-5: principal-scoped-store
Forces: the tenancy decision (tenant-aware design, solo build) means every credential and piece of state must be principal-scoped *now*, because retrofitting isolation later is brutal — but the operational platform (managed secret vault, audit) shouldn't be over-built before the value is proven. Chosen path: one principal-scoped store with at-rest encryption, embedded (SQLite-class, pure-Go — no external dependency) for the single-tenant v1, behind an interface that swaps to a managed store when org-owned multi-tenant arrives. Slack token rotation forces a refresh loop, so the store must persist refresh tokens durably per principal. The hard multi-tenant safety invariant ("principal A never reads B's data/tokens") is a *design-readiness* property today and becomes a verified SPEC INV when feature 0002 commits to multi-tenant. Invalidation trigger: going multi-tenant for colleagues flips this to a managed, audited secret store on org-governed infra — non-negotiable, since it would then hold other people's credentials.

## Decision-6: pluggable-model-provider-with-tiered-routing
Forces: the external-LLM hop is the main data-residency tripwire, the concrete provider is deferred pending a socar-approved/zero-retention route, and cost must stay sane on an always-on service. Chosen path: a provider interface (so the concrete model is a config swap, not a code change) plus tiered model routing — cheap model for the high-volume triage classification, capable model only for the comparatively rare drafting step. RouteLLM-style evidence shows routing the easy majority to a cheap model preserves most quality at a fraction of cost; here it also *minimizes egress*, since most messages never reach the capable model or leave in full. A redaction pass strips obvious secrets/PII before any send. Default v1 = direct provider API with retention/training disabled (acceptable for your *own* data); the in-cloud Bedrock/Vertex route is the upgrade once org-owned. Invalidation trigger: an org policy that forbids external LLM egress entirely forces an in-VPC/self-hosted model behind the same interface before any org data flows.

## Decision-7: drafting-policy-and-voice-profile
Forces: sounding like the principal (`INV-4`) and respecting the plain-register rule (`INV-3`) are quality properties an LLM produces probabilistically, not guarantees. Alternative weighed: fine-tuning a per-user model. Rejected for v1 as far too heavy and data-hungry; few-shot conditioning on exemplars mined from the principal's own recent Slack messages, plus a short style descriptor, gets most of the way at a fraction of the cost and updates continuously as their style drifts. The drafting policy also carries the non-negotiable register/conduct rules (no honorifics toward the principal; professional tone). These outputs are then gated by Decision-8 rather than trusted blindly. Invalidation trigger: if few-shot voice match plateaus below the eval bar, consider light fine-tuning — but only after the eval harness can measure the gap.

## Decision-8: eval-gates-for-quality-invariants
Forces: `INV-4` (voice), `INV-6` (conduct), `INV-7` (grounding) and the routing quality behind `INV-1` cannot be checked by deterministic unit tests — their "correct" is a judgment. Chosen path: an LLM-as-judge offline suite scored against the principal's *real* historical messages (held-out), with the judge itself first aligned to human labels before its scores are trusted (an unaligned judge is just another ungrounded model). Online, the routing decisions are monitored for false-filter rate (a relevant message wrongly ignored — the most damaging error, since it silently violates `INV-1`) and escalation rate (too high = annoying, too low = over-reaching). These metrics, not vibes, drive triage-threshold tuning. Invalidation trigger: judge-human agreement drifting below an acceptable kappa means re-aligning the judge before trusting any gate.

## Decision-9: always-on-single-service-deployment
Forces: one mostly-idle long-running listener, cost-conscious, no public inbound, cloud deliberately undecided. Chosen path: a single Go container that runs anywhere, so the cloud is a deployment-time choice rather than a code dependency; a supervisor restarts and re-establishes the socket on failure to back `INV-8`. Recommended v1 default is a lightweight container host because the workload is bursty-idle and a hyperscaler's always-on minimum is poor value at single-tenant scale — but AWS/GCP become attractive precisely when the in-cloud zero-retention model route (Decision-6) and org ownership arrive together. Invalidation trigger: multi-tenant scale or an org mandate to run on governed infra moves this onto the org's cloud with the matching managed model route.
